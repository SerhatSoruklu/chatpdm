# ChatPDM Concept Review Checklist

## Purpose

This checklist is used to approve, revise, or reject concept drafts before they enter the published ChatPDM concept set.

A concept enters ChatPDM only if it passes structural review.

This checklist functions as a gate. It is not general feedback guidance.

## Review Sections

### 1. Identity and Structure

- [ ] `conceptId` is unique and stable
- [ ] `title` is clean, canonical, and non-ambiguous
- [ ] domain assignment is correct
- [ ] concept is not obviously synonymous with an existing concept

### 2. `shortDefinition`

- [ ] exactly one sentence
- [ ] expresses one atomic relation only
- [ ] defines rather than describes
- [ ] contains no examples
- [ ] contains no hedging language
- [ ] contains no hidden multi-clause logic

### 3. `coreMeaning`

- [ ] adds new structural information
- [ ] explains internal mechanism or governing distinction
- [ ] does not repeat `shortDefinition`
- [ ] contains no examples
- [ ] contains no domain drift

### 4. `fullDefinition`

- [ ] reduces ambiguity
- [ ] follows the intended internal structure
- [ ] remains non-narrative
- [ ] contains no filler
- [ ] contains no teaching tone

### 5. `contexts`

- [ ] each context is distinct
- [ ] contexts do not overlap materially
- [ ] contexts do not repeat `fullDefinition`
- [ ] each context adds domain-level meaning

### 6. `sources`

- [ ] each source has a clear grounding role
- [ ] no decorative sources are present
- [ ] no vague authority references are used

### 7. `relatedConcepts`

- [ ] each relation is directional and meaningful
- [ ] relation type is clear
- [ ] no relation is decorative

### 8. Cross-Concept Consistency

- [ ] concept does not collapse into an existing concept
- [ ] nearby concepts remain clearly separable
- [ ] no contradiction with the current concept set

### 9. Tone Enforcement

- [ ] no chatbot phrasing
- [ ] no rhetorical language
- [ ] no AI-style filler
- [ ] no conversational drift

### 10. Minimality

- [ ] every sentence adds new information
- [ ] no section exists only to sound complete
- [ ] concept is as short as possible without losing structural precision

### 11. Final Gate

- [ ] concept increases explanatory power of the system
- [ ] removing the concept would weaken the system meaningfully
- [ ] concept connects structurally to the current concept set

## Final Review Outcome

### Pass

Use when the concept satisfies the writing standard without structural weakness.

### Revise

Use when the concept is viable but fails one or more correctable checks.

### Reject

Use when the concept is structurally weak, redundant, off-scope, or not separable from the existing concept set.
