Purpose:

- define what counts as canonical truth in ChatPDM
- separate official governed truth from open-source reuse, forks, and local variants
- state the guarantees and non-guarantees attached to the official system

Governing question:

- what counts as canonical truth in ChatPDM, and what bounded guarantees attach to it?
This file owns the public truth boundary.
It does not assign constitutional offices, define deployment sovereignty, or govern trademark and identity representation.

Core statement:

- open code does not mean open truth authority
- ChatPDM canonical truth exists only through the official repository, official releases, official hosted runtime, and official governance chain

## 1. Canonical Boundary

ChatPDM canonical truth exists only when meaning is produced, published, or ratified through the official governance path.

Canonical status depends on all of the following:

- official repository origin
- protected main branch or signed release artifacts
- official hosted runtime or another explicitly declared official trust surface
- governed versions for concepts, concept sets, contracts, matchers, normalizers, and related official truth objects
- traceable governance records for promotion, rollback, freeze, exception, or amendment where relevant

Canonical truth applies to the official meaning layer, including:

- canonical concept packets
- resolve rules and source hierarchy
- refusal behavior and scope boundaries
- official response-contract semantics
- official version declarations
- official governance records that ratify or preserve the truth layer

## 2. Open Code And Official Truth

ChatPDM may be forked under its code license.
That permission does not transfer canonical truth authority.

Forks, mirrors, wrappers, preview deployments, local branches, and experimental builds may reuse code.
They do not become canonical merely by sharing implementation ancestry.

If meaning or governance diverges outside the official path:

- the result is non-canonical by definition
- the divergence must not be presented as official governed truth

## 3. Public Truth Claim Rule

A public claim that something is canonical must tie back to at least one official authority surface:

- the official repository and governed branch history
- an official signed release
- the official hosted runtime
- an official governance record or version declaration

Claims without a traceable official authority path are not canonical claims.
They are local, forked, derived, cached, stale, or experimental claims.

## 4. Guarantees

Inside the official governed runtime, ChatPDM guarantees:

- bounded determinism rather than improvisational answer generation
- source-aware and version-bound handling of active authored concepts
- refusal or constrained resolution when the request exceeds active authored scope
- visible scope boundaries instead of hidden domain broadening
- traceable governance linkage for promoted truth objects and official corrections

The governing rule is:

- refusal over overclaim

## 5. Non-Guarantees

ChatPDM does not guarantee:

- universal completeness
- timeless coverage across all domains
- unbounded authority outside the active authored layer
- that unofficial forks, wrappers, or stale mirrors preserve official integrity conditions
- that local modifications inherit canonical status without official ratification

Bounded determinism is not a promise of omniscience.
It is a promise that the official system will stay inside its governed truth boundary.

## 6. Forks, Mirrors, And Local Variants

Forks are permitted.
Undisclosed equivalence claims are not.

If a fork, wrapper, or local deployment changes:

- canonical concept content
- source priority or admissibility handling
- resolver or refusal behavior
- trust-surface wording
- lifecycle or governance behavior

then the result does not inherit canonical truth authority unless the official project explicitly verifies that status.
Stale, pinned, cached, or locally modified deployments are therefore not automatically canonical merely because they still resemble the upstream system.

## 7. Relationship To Identity Policy

This file defines the official truth boundary.
It does not own trademark, branding, fork-disclosure mechanics, or official-output representation rules.

Those representation and identity rules belong in:

- [TRADEMARK_AND_IDENTITY_POLICY.md](./TRADEMARK_AND_IDENTITY_POLICY.md)

## 8. Closing Statement

ChatPDM guarantees governed determinism, not universal jurisdiction.
Its truth authority is strongest when it stays explicit, versioned, refusal-first, and bounded.
