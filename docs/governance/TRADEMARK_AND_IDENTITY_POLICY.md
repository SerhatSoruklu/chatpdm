# ChatPDM Trademark And Identity Policy

Status: Draft governance doctrine for official identity, fork disclosure, and canonical-output honesty.
Position in governance spine: `09`.
Parent documents:

- [GOVERNANCE_CONSTITUTION.md](./GOVERNANCE_CONSTITUTION.md)
- [CANONICAL_TRUTH_STATEMENT.md](./CANONICAL_TRUTH_STATEMENT.md)

Purpose:

- define the official identity boundary around the ChatPDM name, logo, domains, and officiality claims
- require disclosure when forks, wrappers, mirrors, or derivatives diverge from official governed behavior
- define honest representation rules for official canonical outputs
- document bounded misuse signals and response logic for public confusion about canonical status

Governing question:

- how must ChatPDM identity, officiality claims, and official outputs be represented honestly?
This file owns identity, representation, and disclosure boundaries.
It does not define constitutional truth law, steward powers, or package sovereignty.

Core statement:

- the code may be forked, but the identity of the official truth system remains reserved
- altered systems and altered outputs must not be presented as if they were official canonical ChatPDM

## 1. Official Identity Surfaces

The official identity boundary includes:

- the ChatPDM name
- official logos and marks
- official repository origin
- official domains and hosted services
- official compatibility, continuity, support, and sponsorship claims
- official release origin and official trust-surface wording

These surfaces identify the official governed system.
They are not transferred merely because code is copied or adapted.

## 2. Permitted Nominative Use

Third parties may describe ChatPDM nominatively when doing so honestly.

Permitted use includes:

- naming ChatPDM as the upstream project
- identifying that a system is a fork, wrapper, mirror, or compatibility layer
- referencing official documentation, releases, or governance records for comparison

Prohibited use includes:

- implying official sponsorship, approval, or canonical equivalence without permission
- using names, logos, or domains in a way that collapses fork identity into official identity
- presenting unofficial behavior as the official hosted service

## 3. Fork Disclosure Duty

Any fork, derivative, wrapper, mirror, or downstream package that claims compatibility, continuity, support alignment, or canonical closeness must disclose divergence clearly.

Disclosure must identify, where relevant:

- whether concept content changed
- whether source priority or admissibility changed
- whether resolver, matcher, normalizer, or refusal behavior changed
- whether trust copy, inspect surfaces, or lifecycle gates changed
- the version boundary at which divergence began

Forks are not forbidden.
Undisclosed divergence is forbidden when canonical alignment is implied.

## 4. Official Output Representation

Official canonical outputs may be cited, embedded, or operationalized only when their official origin remains clear.

If an official output is:

- modified
- summarized
- transformed
- cached
- benchmarked
- exported
- screenshotted without context
- used for training or probabilistic marketing claims

then the downstream representation must disclose that it is altered and no longer canonical output in its original official form.

The prohibited boundary is:

- do not relabel altered output as a direct canonical result

## 5. Compatibility And Support Claims

Claims such as `compatible with ChatPDM`, `ChatPDM-aligned`, `official support`, or `canonical equivalent` require a traceable basis.

That basis must identify:

- the upstream version or release claimed
- the scope of compatibility being claimed
- any known divergence
- whether the claim is self-asserted or officially verified

No downstream system may imply canonical equivalence unless the official project has explicitly verified that claim.

## 6. Identity Honesty Boundary

This policy protects the line between:

- official governed origin
- unofficial reuse

It also protects the line between:

- official canonical outputs
- altered, derived, stale, or local outputs

The honesty rule is simple:

- if the identity, origin, or canonical status changed, the disclosure must be visible

Canonical status itself is defined by:

- [CANONICAL_TRUTH_STATEMENT.md](./CANONICAL_TRUTH_STATEMENT.md)

## 7. Appendix: Misuse Signals

The following are misuse signals because they create public confusion about canonical status:

- hidden inference added to an official-looking surface
- refusal behavior softened while official identity is retained
- source grounding or version trace removed
- fork naming or branding designed to imply official sponsorship
- outputs framed as if they came directly from the official runtime after downstream transformation
- compatibility claims made without explicit version basis
- interfaces that make governed resolution look like freeform generative answers

## 8. Appendix: Response Logic

When identity misuse or canonical confusion occurs, the official project should respond by one or more of the following:

- mark the altered system or altered output as non-canonical
- publish a clarification about official status
- freeze or correct misleading official trust claims if confusion affects the hosted service
- route the issue through governance review where doctrine, compatibility, or public trust is affected

This appendix is bounded.
It exists to support identity protection and fork honesty.
It does not create a separate threat-model document.

## 9. Closing Statement

Identity policy is not anti-open-source.
It is the honesty boundary that keeps official ChatPDM truth claims from dissolving into copyable code alone.
